BUILDTOVALUE · DELTA DE DESIGN AG-2 · CONTROLE DE DECISÃO + TOKENS

Uma forma canônica de decisão — e o token que a sustenta

O desenvolvedor inventou, na implementação, o controle de decisão da user task (chips derivados de decisionOptions). Ele tem um irmão que passou por mim: as saídas aprovar/reprovar/escalar do gate de agente (P1). São a mesma ação. Aqui está a forma única que serve aos dois — antes de a AG-3 replicar duas línguas em sete superfícies.

↓ 1 · o modelo · 2 · os dois irmãos · 3 · estados · 4 · o token ■ shared-ui (direto) ■ ESCOPO (passa pelo plano)
1 · O modelo — um controle, opções com intenção. A cor vem da intenção, e só quando ela é conhecida.
A OPÇÃO CANÔNICA
DecisionOption {
  value: string // dado auditado (do gateway)
  label: string // humano, nunca o cru
  intent: 'routing' // padrão
    | 'affirmative' | 'destructive' | 'neutral'
}

Um só componente — mesma forma, interação, seleção e foco em toda parte. O que muda entre user task e gate é a configuração, não a linguagem.

INTENÇÃO → COR (regra)
routing dourado/gate — o padrão honesto. Rota de gateway cujo “mérito” o sistema não conhece.
affirmative verde. Avança/aprova. Só quando o significado é conhecido pelo sistema (gate P1; ou rota anotada).
destructive vermelho. Reprova/encerra caminho. Idem: intenção conhecida.
neutral discreto. Escalar, adiar — sai do fluxo sem julgá-lo.
Isto dissolve o medo de “duas línguas”: a user task usa routing (o dourado que o dev já construiu — ratificado), porque o sistema não sabe qual rota é “boa”. O gate P1 usa cores de intenção porque sabe. Mesma peça; a cor só aparece quando há significado real.

4 · O token — ratifico a cor, corrijo o nome. Item 1 do delta.
HOJE — TRÊS SIGNIFICADOS NUM NOME SÓ
--ui-role-warning-* = #f6edd4 / #6d5115
↳ lint aviso (legítimo)
↳ gate/decisão que roteia ← alheio
↳ tag “pessoal” (classificação) ← alheio

Um chip “PESSOAL” (dado sensível) e um “decisão roteia” (gate) saem idênticos. Quando a AG-3 quiser o gate com preenchimento sólido e o aviso suave, não dá — mexeria em tudo.

PROPOSTA — UMA RAMPA, NOMES POR INTENÇÃO
// primitiva (privada): a rampa dourada
--gold-bg/fg/solid = #f6edd4…
// papéis públicos (por intenção)
--ui-role-warning-* → gold // aviso
--ui-role-gate-*   → gold // decisão/gate
--ui-class-personal → escala própria

Mesma cor hoje (nada muda visualmente agora) — mas nomeados pela intenção, cada um pode divergir depois sem retrofit nas sete superfícies. É o princípio “papel semântico, não hex” da decisão C (F3) e do Atlas E2.

FALTA UM PAPEL — AGENTE (violeta)

O tokens.css tem 5 famílias (success·warning·danger·info + superfícies). Não existe papel de agente/squad. O selo de procedência (E2) e P1–P7 precisam do violeta #5b57b8 / tint #ecebf6. Defini-lo agora em tokens.ts (com o teste de contraste AA) evita que a AG-3 invente um violeta ad-hoc — exatamente como o dourado nasceu.

--ui-role-agent-solid: #5b57b8
ONDE MORA

packages/shared-ui/src/tokens.ts (fonte da verdade, com teste AA) → espelhado em tokens.css. Tudo isto é forma/linguagem → entra direto, sem cerimônia de plano. Só o .decision-option passa a apontar --ui-role-gate-* e ganha o piso de 44px.
