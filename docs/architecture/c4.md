# Arquitetura — C4 Contexto + Container (G-ARQ-3)

> Documentado UMA VEZ aqui; sequências dos 3 fluxos críticos entram na F2
> (avanço+outbox; lock/complete com fencing; submissão de formulário→avanço).

## Nível 1 — Contexto

```
[Usuário de negócio]──┐
[Analista de processo]─┼──▶ (Console BuildToValue) ──▶ (Plataforma BuildToValue) ──▶ [PostgreSQL]
[Operador]────────────┘                                        │
                                                               ├──▶ [Sistemas externos] (jobs http-call/webhook, F2)
                                                               └──▶ [npm @buildtovalue/*] (engine/forms/registry — biblioteca pública)
```

A plataforma é um BPMS multi-tenant: modelagem governada (registry da
biblioteca), execução de instâncias com tarefas humanas + formulários
versionados, jobs, timers e auditoria. O domínio semântico (engine BPMN, forms,
ledger) vive no repo público `danzeroum/bpmn` e entra AQUI só por versões
publicadas exatas (D5).

## Nível 2 — Containers

| Container | Tecnologia | Responsabilidade |
|---|---|---|
| `apps/console` | SPA React (Vite) | 4 rotas (D15/D23): Tarefas·Formulários·Operação·Estúdio; consome a API via SDK gerado do OpenAPI |
| `apps/api` | Fastify + zod | `/v1`; auth (JWT curto + refresh rotativo); tenancy (RLS via `SET LOCAL`); problem+json; rate limit por tenant; OpenAPI servido |
| `apps/worker` | Node | F2: dispatcher da outbox (LISTEN dedicado + poll dinâmico), timers, `JobHandlerRegistry` (handlers FORA de transação, D22) |
| PostgreSQL 16 | — | fonte única: estado (JSONB versionado), variables, outbox (mesma tx do estado, D11), jobs (lease+fencing D12), history; RLS em TUDO (D7); sem particionamento na v1 (D16r) |
| Jaeger (dev) | OTel | tracing request→advance→tx→dispatch→complete (F2) |

Pacotes internos: `config` (env zod), `observability` (pino+redaction testada,
Prometheus), `db` (migrações forward-only, RLS, repositórios, KeyProvider na
F2), `auth` (JWT/RBAC), `api-contracts` (DTOs zod+OpenAPI; `src/jobs/`
reservado ao futuro worker-sdk), `shared-ui` (tokens D25).

**Estilo (G-DAD-3): monólito modular** — api, worker e console deployam
independentes, mas um único domínio e um único banco; escala horizontal por
processo, sem fragmentação prematura.
